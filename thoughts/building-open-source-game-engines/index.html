Building Two Open-Source Game Engines in Six Months (With AI)

29 May 2025

Have you ever tried to embed a chess game on a website? If you have, you probably reached for one of the established libraries — Chessboard.js, chess.js, Lichess's board component. They work. But what if you wanted Atomic Chess? Or Racing Kings? Or a variant where pieces can clone themselves? Suddenly you're forking entire codebases and fighting against architectures designed for one game.

That frustration is what led me to build Moddable Chess Engine — a modular engine that treats variants as first-class citizens. And once I had the architecture patterns right, the same thinking produced Moddable Hexmaps — a framework where any hex-based board game can be generated, edited, and shared with a URL.

Both shipped to production in 16 days. Both are MIT-licensed. And both were built almost entirely using AI-augmented engineering workflows.

The Chess Engine: 70 Variants, Zero Dependencies

The Moddable Chess Engine supports 70 playable variants on boards from 4x8 to 12x8. It has a multi-depth AI with Web Worker threading, iterative deepening, transposition tables, quiescence search, opening books, and variant-aware evaluation. It runs entirely client-side — pure JavaScript, no build step, no dependencies. You can embed it with a single iframe tag.

But the interesting part isn't what it does — it's how it's designed to be extended. Every variant is a self-contained plugin file. The engine exposes a tiered hook system (Tier 1: rule overrides, Tier 2: engine extensions, Tier 3: effects, mutations, and action moves). Multiple developer guides walk through each complexity level — from basic rule variants to Crazyhouse-style piece drops to Poison Chess effects.

This is a DevRel problem as much as an engineering one. The question isn't "can I make the engine support more variants?" — it's "can I make the extension interface so clear that a community member could add one without asking me?" That meant investing heavily in documentation, tiered examples, and a plugin architecture that fails loudly when you get the interface wrong.

  • 70 playable variants — from Atomic Chess to Medusa Chess
  • PostMessage embed API — control variants and themes without iframe reload
  • Tiered plugin system — Tier 1/2/3 variants with registerVariant()
  • AI with opening books — Web Worker search, variant-specific evaluators, multiple difficulty levels
  • Developer guides — Build a Variant, Crazyhouse (Tier 3), Poison Chess (effects)

The Hex Map Framework: Infinite Seeded Worlds

Moddable Hexmaps solves a different problem: board game designers need maps, but designing them manually is tedious and sharing them is friction-heavy. What if every map had a URL? And what if that URL could be deterministically regenerated from a seed string?

The framework generates hex grids via Canvas — flat-top or pointy-top, at any ring count. It uses seeded RNG so the same seed always produces the same map. Multiple terrain palettes support different games (Nukes gets volcanic landscapes, Talisman gets fantasy biomes). Maps export to JSON. Maps import from JSON. Maps embed in iframes. Everything is controlled via URL parameters.

This means sharing a map is sharing a URL. Reproducibility is guaranteed by the seed. And because everything renders client-side, there's no server cost regardless of how many maps people generate.

  • Seeded procedural generation — same seed = same map, always
  • 4 games supported — Nukes, Talisman, Twilight Imperium, Colony
  • URL parameter API — ?theme=nukes&rings=4&seed=volcano
  • Click-to-edit — manual overrides on any generated hex
  • JSON export/import — save, share, and reload board states

The AI-Augmented Part

Both engines were built using LLM-driven development as the core methodology. Not "AI wrote my code" — but AI as a pair programmer operating at the architecture level. Designing hook systems, catching edge cases across 70 variants, writing documentation that developers can actually follow, identifying which abstractions pay for themselves and which are premature.

The velocity difference is real. A solo founder shouldn't be able to ship two engines, four games, nine mods, a tools platform, and a 190-page rulebook pipeline in 16 days. But when your pair programmer has read every programming book ever written and never gets tired, the bottleneck shifts from "can I build this?" to "should I build this?"

That's the actual lesson. The engines are the proof that the methodology works. The methodology is the thing worth talking about.

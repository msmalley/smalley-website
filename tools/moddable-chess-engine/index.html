The Problem

Existing chess libraries — Chessboard.js, chess.js, Lichess components — are designed for standard chess. If you want Atomic Chess, Racing Kings, or any variant with non-standard rules, you're forking entire codebases and fighting architectures that assume one set of rules. There was no engine that treated chess variants as a first-class extension point.

Architecture

The engine separates concerns into layers that can be independently replaced:

  • Board geometry — supports boards from 4x8 to 12x8 with configurable square properties
  • Move generation — base engine handles standard moves; variants override via hooks for custom piece behaviour, capture rules, and victory conditions
  • AI opponent — minimax with alpha-beta pruning, configurable depth, piece-value evaluation that adapts to variant rules
  • Rendering — pure Canvas, no DOM manipulation during gameplay. Responsive, touch-friendly, works in iframes
  • Plugin interface — each variant is a single file calling registerVariant() with its rule overrides

The Developer Experience

This is a DevRel problem as much as an engineering one. The question isn't "can the engine support more variants?" — it's "can someone add one without understanding the engine internals?"

The "Build a Variant" guide walks a developer through creating a new variant from scratch. The plugin interface fails loudly with clear error messages when the contract is violated. Example variants ship as reference implementations at increasing complexity levels.

Embed API

Any variant can be embedded on any website with a single iframe. URL parameters control everything:

  • ?variant=atomic — select the variant
  • ?ai=black&depth=3 — configure AI opponent
  • ?embed=1 — minimal chrome for embedding
  • ?fen=... — start from a specific position
